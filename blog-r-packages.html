From Scripts to R Packages: Refactoring the DeGIR Pipeline

How I turned a collection of analysis scripts into a proper R package during my internship at HealthTwiSt GmbH.

Why Package Your Code?

When I first started working on the DeGIR registry pipeline at HealthTwiSt, the codebase was a collection of R scripts. There was a 01_import.R, a 02_clean.R, a 03_analysis.R, and so on. Each script sourced the previous one with source("02_clean.R"). It worked, in the sense that it produced correct output most of the time.

But "most of the time" is not good enough when 300+ radiology clinics across Germany depend on your pipeline running correctly every quarter. Here is what went wrong with the script-based approach:

  • No versioning — When something broke, there was no way to tell which version of which script was running on the production server versus the development machine.
  • No testing — Changes to one helper function could silently break downstream calculations. We only discovered this when someone noticed the numbers looked wrong in a quarterly report.
  • Namespace pollution — Every script dumped its variables and functions into the global environment. A helper function called clean() in one script could collide with clean() from another package.
  • No documentation — "What does this function expect as input?" was a question answered by reading the source code, if you could find the right file.

An R package solves all of these problems. It gives you a standard structure, automated documentation, unit tests, namespace isolation, and semantic versioning. The overhead of setting one up is maybe an afternoon — the return on that investment is enormous.

Package Structure

The standard R package layout is well-defined and enforced by R CMD check. Here is what our package directory looks like:

degirtools/
├── DESCRIPTION          # Package metadata, version, dependencies
├── NAMESPACE            # Generated by roxygen2 — do not edit manually
├── LICENSE
├── R/
│   ├── import.R         # Functions for reading raw registry exports
│   ├── clean.R          # Data cleaning and validation functions
│   ├── transform.R      # Derived variables (age, complication flags, etc.)
│   ├── summarise.R      # Aggregation and summary table functions
│   └── utils.R          # Internal helper functions
├── man/                 # Generated documentation (from roxygen2)
│   ├── import_registry.Rd
│   ├── clean_dates.Rd
│   └── ...
├── tests/
│   ├── testthat.R       # Test runner
│   └── testthat/
│       ├── test-clean.R
│       ├── test-transform.R
│       └── test-summarise.R
├── inst/
│   └── extdata/         # Sample data for examples and tests
└── vignettes/
    └── pipeline-overview.Rmd

Setting this up from scratch is straightforward with usethis:

# Create the package skeleton
usethis::create_package("degirtools")

# Set up testing infrastructure
usethis::use_testthat()

# Add a license
usethis::use_mit_license("HealthTwiSt GmbH")

# Declare dependencies
usethis::use_package("dplyr")
usethis::use_package("tidyr")
usethis::use_package("lubridate")
usethis::use_package("readr")

The DESCRIPTION file is the heart of the package. It declares the package name, version, author, description, and — critically — its dependencies. When someone installs your package, R knows exactly which other packages it needs:

Documentation with roxygen2

In the old script-based workflow, documentation meant comments at the top of a file that were perpetually out of date. With roxygen2, documentation lives directly above each function, and it generates the standard R help pages automatically.

Here is a documented function from our package:

#' Calculate patient age at time of procedure
#'
#' Computes age in years from birth date and procedure date,
#' using 365.25 days per year to account for leap years.
#' Returns NA if either date is missing.
#'
#' @param birth_date A Date vector of patient birth dates.
#' @param procedure_date A Date vector of procedure dates.
#'
#' @return A numeric vector of ages in years.
#'
#' @export
#'
#' @examples
#' calculate_age(
#'   as.Date("1990-01-15"),
#'   as.Date("2024-06-01")
#' )
calculate_age <- function(birth_date, procedure_date) {
  if (!inherits(birth_date, "Date") || !inherits(procedure_date, "Date")) {
    rlang::abort("Both arguments must be Date objects.")
  }
  as.numeric(difftime(procedure_date, birth_date, units = "days")) / 365.25
}

The key roxygen2 tags:

  • @param — Documents each function argument with its expected type and meaning.
  • @return — Describes what the function returns.
  • @export — Makes the function available to users of the package. Without this tag, the function is internal only.
  • @examples — Provides runnable examples that are checked during R CMD check.

Running devtools::document() reads all the roxygen2 comments, generates the .Rd files in man/, and updates the NAMESPACE file with your exported functions. It takes about two seconds and keeps everything in sync.

Testing with testthat

Unit tests were the single biggest improvement in our workflow. Before tests, the only way to verify the pipeline was to run the entire thing and visually inspect the output. Now, every function has tests that run in seconds and catch regressions immediately.

Here is a test file for the age calculation function:

# tests/testthat/test-transform.R

test_that("age calculation returns correct values", {
  age <- calculate_age(
    as.Date("1990-01-01"),
    as.Date("2024-01-01")
  )
  expect_equal(age, 34, tolerance = 0.01)
})

test_that("age calculation handles leap year birthdays", {
  age <- calculate_age(
    as.Date("2000-02-29"),
    as.Date("2024-02-29")
  )
  expect_equal(age, 24, tolerance = 0.01)
})

test_that("age calculation returns NA for missing dates", {
  age <- calculate_age(
    as.Date(NA),
    as.Date("2024-01-01")
  )
  expect_true(is.na(age))
})

test_that("age calculation rejects non-Date input", {
  expect_error(
    calculate_age("1990-01-01", "2024-01-01"),
    "must be Date objects"
  )
})

The development workflow becomes a tight loop:

# Write or modify a function
# Write or modify its tests
devtools::test()        # Run all tests (~5 seconds)
devtools::document()    # Regenerate documentation
devtools::check()       # Full R CMD check (before release)

devtools::test() runs all tests and reports failures immediately. devtools::check() is the comprehensive check that CRAN would run — it checks documentation, examples, tests, and package structure. We run check() before every quarterly release; we run test() dozens of times a day during development.

The investment in packaging paid off within the first quarter. A colleague modified a date parsing function without realizing it affected the age calculation downstream. The test suite caught it in seconds, before the change ever reached production. That single catch justified the entire effort.

Interview Quick Review

What is R package development?

The practice of organizing R code into a standardized package structure with formal documentation, unit tests, namespace management, and dependency declarations, using tools like devtools, roxygen2, and testthat.

When would you use it?

When your R code is used repeatedly (especially by others or in production), when you need reproducibility guarantees, or when a project grows beyond a handful of scripts and requires proper testing and documentation.

Key function to remember:

devtools::document()  # Generate docs from roxygen2 comments
devtools::test()      # Run all unit tests
devtools::check()     # Full R CMD check

Common interview question:

"What are the benefits of packaging R code vs. sourcing scripts?"

Packages provide namespace management (avoiding function name conflicts between packages), automated documentation generation via roxygen2, unit testing with testthat, semantic version control, explicit dependency declaration in the DESCRIPTION file, and easy distribution via install.packages() or remotes::install_github(). Scripts with source() offer none of these guarantees and quickly become unmaintainable as projects grow.

One thing to remember:

The three-command workflow — devtools::document(), devtools::test(), devtools::check() — covers documentation, testing, and validation. Learn these three and you can develop R packages effectively.
