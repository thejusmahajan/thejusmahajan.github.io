Building Clinical Dashboards with R Shiny

Lessons from building an interactive quality metrics dashboard for the German DeGIR interventional radiology registry.

Why Shiny for Clinical Data?

For years, the DeGIR registry reporting workflow looked like this: run R scripts, generate static tables and plots, paste them into a Word document, export to PDF, email to clinics. Every quarter, the same manual process. Every quarter, someone would ask, "Can we filter this by region?" or "What if we look at just the last two years?" and the answer was always, "I will regenerate the report and send you a new PDF."

Shiny changed that entirely. Instead of static reports, we built an interactive web application that lets clinicians and registry administrators explore the data themselves. Filter by clinic, time period, procedure type, patient demographics — all in real time, all without writing a single line of code on their end.

The key advantage of Shiny over other dashboard tools (Tableau, Power BI) is that it is native R. The same dplyr pipelines, the same ggplot2 visualizations, the same statistical models that we use in our analysis scripts can be embedded directly into the dashboard. There is no translation layer, no data export step, no second tool to learn. If you can write R, you can build a Shiny app.

The dashboard we built at HealthTwiSt now serves quality metrics — procedure volumes, complication rates, radiation dose distributions, outcome benchmarks — to over 300 German radiology clinics. Each clinic can log in and see their own performance compared to national benchmarks, updated every quarter when we refresh the underlying data.

The Reactive Programming Model

Shiny's programming model is different from standard R scripting, and this is where most people struggle initially. In a script, code runs top to bottom, once. In Shiny, code runs reactively — when an input changes, only the parts of the application that depend on that input are re-executed.

The core structure of every Shiny app has two parts: the UI (what the user sees) and the server (the logic behind it):

library(shiny)

ui <- fluidPage(
  titlePanel("DeGIR Quality Metrics"),

  sidebarLayout(
    sidebarPanel(
      selectInput(
        "clinic", "Select Clinic:",
        choices = clinic_list
      ),
      dateRangeInput(
        "date_range", "Time Period:",
        start = "2023-01-01", end = Sys.Date()
      ),
      selectInput(
        "metric", "Quality Metric:",
        choices = c("Procedure Volume", "Complication Rate",
                    "Radiation Dose", "Fluoroscopy Time")
      )
    ),

    mainPanel(
      plotOutput("trend_plot"),
      tableOutput("summary_table")
    )
  )
)

server <- function(input, output, session) {

  # Reactive expression: filtered data
  # This re-runs ONLY when input$clinic or input$date_range changes
  filtered_data <- reactive({
    registry_data %>%
      filter(
        clinic_id == input$clinic,
        procedure_date >= input$date_range[1],
        procedure_date <= input$date_range[2]
      )
  })

  # Render the trend plot
  # This re-runs when filtered_data() or input$metric changes
  output$trend_plot <- renderPlot({
    df <- filtered_data()

    ggplot(df, aes(x = procedure_date, y = .data[[input$metric]])) +
      geom_line(color = "#0284c7", linewidth = 1) +
      geom_smooth(method = "loess", se = TRUE, alpha = 0.2) +
      labs(title = paste(input$metric, "Over Time"),
           x = "Date", y = input$metric) +
      theme_minimal(base_size = 14)
  })

  # Render the summary table
  output$summary_table <- renderTable({
    filtered_data() %>%
      summarise(
        `Total Procedures` = n(),
        `Mean Age` = round(mean(age, na.rm = TRUE), 1),
        `Complication Rate` = paste0(
          round(mean(had_complication, na.rm = TRUE) * 100, 1), "%"
        )
      )
  })
}

shinyApp(ui = ui, server = server)

The key concepts to understand:

  • reactive({}) — Creates a reactive expression that caches its result. It only re-computes when its reactive dependencies (like input$clinic) change. Use this for expensive computations that multiple outputs share, like filtering a large dataset.
  • renderPlot({}) / renderTable({}) — Reactive endpoints that produce output. They automatically track which reactive values they use and re-render when those values change.
  • observe({}) — Like reactive({}), but for side effects (logging, updating UI elements, writing to a database). It runs whenever its dependencies change but does not return a value.

The mental model that helped me most: think of it as a spreadsheet. Cell B1 contains a formula that references A1. When you change A1, B1 automatically updates. Shiny works the same way — input$clinic is cell A1, filtered_data() is B1, and output$trend_plot is C1. Change the clinic dropdown, and the entire chain updates automatically.

Modern UI with bslib

The default Shiny UI looks dated. It uses Bootstrap 3 by default, and the stock widgets are functional but not particularly attractive. The bslib package modernizes everything by providing Bootstrap 5 components and a theming system.

Here is how we structured the dashboard layout using bslib:

library(bslib)

ui <- page_sidebar(
  title = "DeGIR Quality Dashboard",
  theme = bs_theme(
    bootswatch = "flatly",
    primary = "#0284c7",
    base_font = font_google("Inter"),
    heading_font = font_google("Inter")
  ),

  sidebar = sidebar(
    title = "Filters",
    selectInput("clinic", "Clinic:", choices = clinic_list),
    dateRangeInput("date_range", "Period:",
                   start = "2023-01-01", end = Sys.Date()),
    hr(),
    checkboxGroupInput("procedures", "Procedure Types:",
                       choices = procedure_types,
                       selected = procedure_types)
  ),

  # KPI row using value boxes
  layout_columns(
    value_box(
      title = "Total Procedures",
      value = textOutput("n_procedures"),
      showcase = bsicons::bs_icon("clipboard2-pulse"),
      theme = "primary"
    ),
    value_box(
      title = "Complication Rate",
      value = textOutput("complication_rate"),
      showcase = bsicons::bs_icon("exclamation-triangle"),
      theme = "warning"
    ),
    value_box(
      title = "Mean Fluoroscopy Time",
      value = textOutput("mean_fluoro"),
      showcase = bsicons::bs_icon("stopwatch"),
      theme = "info"
    ),
    col_widths = c(4, 4, 4)
  ),

  # Main content cards
  layout_columns(
    card(
      card_header("Procedure Volume Trend"),
      plotOutput("volume_plot", height = "350px")
    ),
    card(
      card_header("Complication Rate by Type"),
      plotOutput("complication_plot", height = "350px")
    ),
    col_widths = c(6, 6)
  ),

  card(
    card_header("Detailed Procedure Data"),
    DT::dataTableOutput("detail_table")
  )
)

The value_box() components are particularly useful for clinical dashboards. Clinicians want to see their key metrics at a glance — total volume, complication rate, average dose — before diving into detailed charts. The card() component provides a clean container with a header, and layout_columns() handles the responsive grid layout.

The bs_theme() function lets you customize the entire look and feel. We used the "flatly" Bootswatch theme as a base, then overrode the primary color to match the DeGIR branding. The font_google() function pulls in web fonts directly.

Interactive Tables and Plots

Two packages dramatically improve the interactivity of a Shiny dashboard: DT for tables and plotly for plots.

DT::datatable() replaces the basic renderTable() with a fully interactive table widget:

output$detail_table <- DT::renderDataTable({
  filtered_data() %>%
    select(
      patient_id, procedure_date, procedure_type,
      age, sex, complications, radiation_dose_mGy
    ) %>%
    DT::datatable(
      options = list(
        pageLength = 15,
        dom = "Bfrtip",         # B = Buttons, f = filter, r = processing,
                                # t = table, i = info, p = pagination
        buttons = c("csv", "excel", "pdf"),
        scrollX = TRUE
      ),
      extensions = "Buttons",
      filter = "top",           # Column-level filters
      rownames = FALSE,
      class = "compact stripe hover"
    )
})

This gives the user a searchable, sortable table with column-level filters and download buttons for CSV, Excel, and PDF. Clinicians loved the download buttons — they could filter the data to exactly what they needed, then export it for their own reports.

plotly::ggplotly() takes any ggplot2 plot and makes it interactive with hover tooltips, zoom, and pan:

output$volume_plot <- renderPlotly({
  p <- filtered_data() %>%
    count(month = floor_date(procedure_date, "month")) %>%
    ggplot(aes(x = month, y = n,
               text = paste("Month:", format(month, "%B %Y"),
                            "<br>Procedures:", n))) +
    geom_col(fill = "#0284c7", alpha = 0.8) +
    geom_smooth(method = "loess", se = FALSE,
                color = "#f97316", linewidth = 1) +
    labs(x = NULL, y = "Number of Procedures") +
    theme_minimal(base_size = 13)

  ggplotly(p, tooltip = "text") %>%
    layout(hoverlabel = list(bgcolor = "white")) %>%
    config(displayModeBar = TRUE,
           modeBarButtonsToRemove = c("lasso2d", "select2d"))
})

The text aesthetic in ggplot2 combined with tooltip = "text" in ggplotly() gives you full control over what appears in the hover tooltip. This is much cleaner than the default plotly tooltips, which tend to dump all aesthetics into the tooltip regardless of whether they are useful.

One practical tip: ggplotly() does not handle every ggplot2 feature perfectly. Complex annotations, custom themes, and certain geoms can render oddly. For critical visualizations where the layout needs to be pixel-perfect, I sometimes use plotly's native R API directly instead of converting from ggplot2. But for 90% of dashboard plots, ggplotly() works beautifully with minimal effort.

Interview Quick Review

What is Shiny?

Shiny is an R package for building interactive web applications directly from R, without requiring HTML, CSS, or JavaScript knowledge. It uses a reactive programming model where outputs automatically update when inputs change.

When would you use it?

When stakeholders need to interactively explore data rather than receive static reports — especially for dashboards, data exploration tools, and applications where users need to filter, drill down, or customize visualizations on the fly.

Key function to remember:

shinyApp(
  ui = fluidPage(
    selectInput("x", "Choose:", choices),
    plotOutput("plot")
  ),
  server = function(input, output) {
    output$plot <- renderPlot({ ... })
  }
)

Common interview question:

"What is reactivity in Shiny?"

Reactivity is Shiny's automatic dependency tracking system. When a reactive input changes (like a dropdown selection), only the outputs and reactive expressions that depend on it are re-executed. This works like a spreadsheet: changing one cell automatically updates all dependent formulas, but cells that do not reference the changed cell remain untouched. The three building blocks are reactive sources (input$*), reactive conductors (reactive({})), and reactive endpoints (render*({})).

One thing to remember:

Use reactive({}) for shared computations that return a value (like filtering a dataset used by multiple outputs), and observe({}) for side effects (like logging or updating UI elements). Mixing these up is the most common source of bugs in Shiny applications.
