HPC Job Management: A Practical SLURM Guide

Everything I wish I knew about SLURM when I started running ocean simulations on the DKRZ supercomputer.

When I joined the marine ecosystem modeling group at the University of Hamburg, my first task was to run a suite of ICON-O ocean simulations on Levante, the supercomputer at the German Climate Computing Centre (DKRZ). I had used Linux extensively during my PhD in astrochemistry, but I had never worked with a job scheduler before. I naively tried to run my model interactively on a login node and was promptly (and rightly) sent a warning email by the system administrators. That was my introduction to SLURM.

SLURM (Simple Linux Utility for Resource Management) is the job scheduler that governs access to computational resources on most modern HPC clusters. You do not run programs directly; instead, you submit job scripts that describe what resources you need and what commands to execute. SLURM then queues your job and dispatches it to available compute nodes when your turn comes. This post covers the practical patterns I use daily.

SLURM Basics: sbatch, squeue, scancel

The fundamental workflow is: write a batch script, submit it with sbatch, monitor it with squeue, and cancel it with scancel if something goes wrong. Here is a minimal but realistic batch script for running an ocean model simulation:

#!/bin/bash
#SBATCH --job-name=icon_ocean_run
#SBATCH --partition=compute
#SBATCH --account=bm1344
#SBATCH --nodes=4
#SBATCH --ntasks-per-node=128
#SBATCH --time=08:00:00
#SBATCH --mem=0
#SBATCH --output=logs/icon_%j.out
#SBATCH --error=logs/icon_%j.err
#SBATCH --mail-type=END,FAIL
#SBATCH --mail-user=thejus.mahajan@uni-hamburg.de

# Load required modules
module load intel-oneapi-mpi/2021.10.0
module load netcdf-c/4.9.2
module load cdo/2.3.0

# Change to the experiment directory
cd /work/bm1344/experiments/north_atlantic_025

# Run the model using MPI across all allocated tasks
srun ./icon_ocean

Let me break down the key directives:

  • --partition=compute selects which pool of nodes to use. On Levante, compute is the standard partition; gpu would be for GPU-accelerated workloads.
  • --nodes=4 and --ntasks-per-node=128 request 4 nodes with 128 MPI tasks each, for a total of 512 MPI ranks. This is how parallel ocean models scale across hundreds of processors.
  • --time=08:00:00 sets a wall-clock limit of 8 hours. If your job exceeds this, SLURM kills it. Always estimate generously but not absurdly -- over-requesting time reduces your scheduling priority.
  • --mem=0 is a DKRZ-specific convention meaning "use all available memory on the node." On other clusters, you might specify --mem=240G explicitly.
  • %j in the output filename expands to the job ID, so each run gets its own log file.

Once the script is ready, the workflow looks like this:

# Submit the job
$ sbatch run_icon.sh
Submitted batch job 12345678

# Check the queue -- see all your running/pending jobs
$ squeue --me
  JOBID  PARTITION     NAME      USER  ST   TIME  NODES  NODELIST(REASON)
12345678   compute  icon_oc  mahajan   R   1:23:45   4  l10[234-237]

# If something is wrong, cancel it
$ scancel 12345678

# Check status of a specific job (including completed jobs)
$ scontrol show job 12345678

A habit I developed early: always check your job's first few lines of output shortly after it starts running. Many configuration errors (wrong file paths, missing modules) cause the job to fail within the first minute, but if you do not check, it might sit in "running" state consuming your allocation while writing error messages to the log file.

Array Jobs for Parameter Sweeps

This is the SLURM feature that saved me the most time. During my postdoc, I needed to run sensitivity experiments: the same ocean model configuration with systematically varied parameters (e.g., different values of vertical mixing coefficients, or different atmospheric forcing datasets). Instead of writing 100 separate submission scripts, SLURM's array jobs let you submit a single script that spawns multiple independent tasks:

#!/bin/bash
#SBATCH --job-name=sensitivity
#SBATCH --partition=compute
#SBATCH --account=bm1344
#SBATCH --nodes=1
#SBATCH --ntasks=64
#SBATCH --time=02:00:00
#SBATCH --mem=120G
#SBATCH --array=1-100
#SBATCH --output=logs/sensitivity_%A_%a.out

# %A = master job ID, %a = array task index

# Read the parameter value for this specific task
# params.txt has one parameter set per line
MIXING_COEFF=$(sed -n "${SLURM_ARRAY_TASK_ID}p" params.txt | cut -f1)
WIND_SCALE=$(sed -n "${SLURM_ARRAY_TASK_ID}p" params.txt | cut -f2)

echo "Task ${SLURM_ARRAY_TASK_ID}: mixing=${MIXING_COEFF}, wind_scale=${WIND_SCALE}"

# Create a unique experiment directory
EXPDIR="/work/bm1344/sensitivity/run_${SLURM_ARRAY_TASK_ID}"
mkdir -p ${EXPDIR}

# Generate the namelist with these specific parameters
sed -e "s/@MIXING_COEFF@/${MIXING_COEFF}/" \
    -e "s/@WIND_SCALE@/${WIND_SCALE}/" \
    namelist_template.nml > ${EXPDIR}/namelist.nml

# Run the model
cd ${EXPDIR}
srun ./icon_ocean

The --array=1-100 directive tells SLURM to launch 100 independent tasks. Each task receives its own $SLURM_ARRAY_TASK_ID (an integer from 1 to 100), which it uses to look up the corresponding parameters from a file. The tasks are scheduled independently, so they can run on different nodes at different times -- SLURM handles all the queuing.

A few practical tips I learned the hard way:

  • Use --array=1-100%20 to limit concurrent tasks to 20 at a time. This prevents you from consuming the entire cluster and annoying your colleagues.
  • If some tasks fail, you can resubmit only the failed ones: --array=3,17,42,89 to run specific task IDs.
  • Always use %A_%a (not %j) in output filenames for array jobs, so each task gets a separate log file.

Resource Management Tips

One of the most common mistakes I see (and made myself early on) is blindly requesting maximum resources for every job. On shared clusters, this wastes allocation budget, increases queue wait times, and is generally antisocial. SLURM provides tools to help you right-size your requests.

The most useful command is sacct, which shows the actual resource consumption of completed jobs:

# Check how much memory and time a completed job actually used
$ sacct -j 12345678 --format=JobID,JobName,Elapsed,MaxRSS,MaxVMSize,State
      JobID    JobName    Elapsed     MaxRSS  MaxVMSize      State
------------ ---------- ---------- ---------- ---------- ----------
12345678     icon_oc+   05:42:31                         COMPLETED
12345678.0        srun  05:42:28   42312456K  98765432K  COMPLETED

# MaxRSS is peak resident memory: ~42 GB out of the 240 GB available per node
# Elapsed is 5h42m out of the 8h requested

From this output, I can see that my job used about 42 GB of memory (out of 240 GB available) and finished in under 6 hours (out of 8 hours requested). For the next run, I could safely reduce the memory request and tighten the time limit. This matters because many clusters use a "fairshare" scheduling policy where your priority decreases based on the resources you consume -- requesting less means your future jobs start sooner.

Another essential pattern is job chaining with dependencies. Ocean model simulations often need to be run in segments: first a spinup phase, then the production run, then a post-processing step. Rather than babysitting each stage, you can chain them:

# Submit the spinup job
$ SPINUP_ID=$(sbatch --parsable run_spinup.sh)
echo "Spinup job: ${SPINUP_ID}"

# Submit the production run -- only starts after spinup succeeds
$ PROD_ID=$(sbatch --parsable --dependency=afterok:${SPINUP_ID} run_production.sh)
echo "Production job: ${PROD_ID}"

# Submit post-processing -- only starts after production succeeds
$ sbatch --dependency=afterok:${PROD_ID} run_postprocess.sh

# Check the dependency chain
$ squeue --me
  JOBID  PARTITION     NAME      USER  ST  TIME  NODES  NODELIST(REASON)
1234001   compute  spinup   mahajan   R   0:45   4  l10[234-237]
1234002   compute  product  mahajan  PD   0:00   8  (Dependency)
1234003   compute  postpro  mahajan  PD   0:00   1  (Dependency)

The --dependency=afterok:JOBID flag means "start this job only if the specified job completed successfully (exit code 0)." If the spinup fails, the downstream jobs will never start, preventing you from wasting compute hours on invalid input. There is also afterany (start regardless of exit status) and afternotok (start only if the dependency failed -- useful for automated error-recovery scripts).

I typically wrap this pattern in a small Bash script that submits the entire pipeline at once. This way, I can submit a multi-stage experiment on Friday evening and come back Monday to find all the results waiting, with each stage having correctly depended on the previous one.

Interview Quick Review

What is SLURM?

SLURM (Simple Linux Utility for Resource Management) is a job scheduler and resource manager used on most HPC clusters. It queues, schedules, and manages batch jobs across hundreds or thousands of compute nodes.

When would you use it?

Whenever you need to run computationally intensive tasks on a shared HPC cluster -- model simulations, large-scale data processing, parameter sweeps, or any workload that requires more resources than a single workstation provides.

Key function to remember:

sbatch --array=1-100 run_model.sh

Common interview question:

"How would you run 1000 independent analyses on an HPC cluster?"

Use SLURM array jobs with the --array=1-1000 flag. Each array task receives a unique $SLURM_ARRAY_TASK_ID environment variable, which you use to index into a parameter file or select input data. SLURM schedules each task independently, so they can run in parallel across available nodes. Use --array=1-1000%50 to limit concurrency and avoid monopolizing the cluster.

One thing to remember:

Always check actual resource usage with sacct after your first few runs, then adjust your #SBATCH requests accordingly. Over-requesting memory and time wastes your allocation budget and lowers your scheduling priority under fairshare policies.
