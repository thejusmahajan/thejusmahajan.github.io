Clinical Data Wrangling with dplyr and tidyr

Real-world lessons from cleaning messy clinical registry data — the operations I use every single day.

The Reality of Clinical Data

If you have ever opened a freshly exported clinical dataset and found it perfectly structured, with consistent date formats, no duplicates, and every missing value neatly coded as NA — I envy you. In my experience working with the German DeGIR interventional radiology registry, the reality is far messier.

Here is what a typical raw export looks like in practice:

  • Inconsistent date formats — Some clinics submit dates as DD.MM.YYYY (the German standard), others as YYYY-MM-DD, and occasionally you get MM/DD/YYYY from a system that was configured for US locale.
  • Duplicated patient records — The same procedure appearing twice because a clinic submitted corrections without flagging the original entry for deletion.
  • Creative missing value encoding — Missing values coded as -999, -1, the string "N/A", "n.a.", "nicht angegeben", or simply an empty string that R reads as "" rather than NA.
  • Wide-format tables — Complication data spread across columns like complication_1, complication_2, complication_3, when what you actually need for analysis is a long-format table with one row per complication.

The tidyverse — specifically dplyr and tidyr — is how I handle all of this. Not because it is trendy, but because the pipe-based syntax maps directly onto how you think about data transformations: "take this dataset, then filter it, then create new columns, then group and summarize."

Essential dplyr Operations

The core of any clinical data pipeline is a chain of dplyr verbs. Here is a realistic pipeline that I might run to produce a quarterly summary of procedure volumes and patient demographics by clinic:

library(dplyr)

quarterly_summary <- registry_data %>%
  # Remove rows where the procedure date is missing entirely
  filter(!is.na(procedure_date)) %>%

  # Standardize the date column (handles mixed formats)
  mutate(
    procedure_date = lubridate::parse_date_time(
      procedure_date, orders = c("dmy", "ymd", "mdy")
    ),
    birth_date = lubridate::parse_date_time(
      birth_date, orders = c("dmy", "ymd", "mdy")
    )
  ) %>%

  # Calculate age at time of procedure
  mutate(
    age = as.numeric(
      difftime(procedure_date, birth_date, units = "days")
    ) / 365.25,
    year = lubridate::year(procedure_date),
    quarter = lubridate::quarter(procedure_date)
  ) %>%

  # Exclude biologically implausible ages
  filter(age >= 0, age <= 120) %>%

  # Summarize by clinic, year, and quarter
  group_by(clinic_id, year, quarter) %>%
  summarise(
    n_procedures = n(),
    mean_age = mean(age, na.rm = TRUE),
    median_age = median(age, na.rm = TRUE),
    pct_female = mean(sex == "F", na.rm = TRUE) * 100,
    .groups = "drop"
  )

Let me walk through the key verbs:

  • filter() — Keeps only rows that match a condition. The first filter removes records without procedure dates; the second removes implausible ages. In clinical data, you always want to be explicit about what you are excluding and why.
  • mutate() — Creates or modifies columns. Here we parse dates into a consistent format, compute patient age, and extract year and quarter from the procedure date.
  • group_by() — Defines the grouping structure for subsequent operations. After this call, every summarise, mutate, or filter operates within each group independently.
  • summarise() — Collapses each group into a single summary row. The .groups = "drop" argument avoids the common gotcha where grouped tibbles cause unexpected behavior downstream.

One thing I have learned the hard way: always use na.rm = TRUE in summary functions when working with clinical data. A single NA in your age column will make mean(age) return NA for the entire group, and you will spend twenty minutes wondering why your summary table is full of missing values.

Reshaping with tidyr

Clinical registries love wide-format data. It makes sense from a data entry perspective — one row per patient, with columns for each possible complication. But for analysis, you almost always need long format.

Consider a dataset where complications are stored like this:

# Wide format (as received from the registry)
# patient_id | complication_1 | complication_2 | complication_3
# 001        | bleeding       | infection      | NA
# 002        | NA             | NA             | NA
# 003        | thrombosis     | NA             | NA

To analyze complication rates, you need to pivot this to long format:

complications_long <- registry_data %>%
  pivot_longer(
    cols = starts_with("complication_"),
    names_to = "complication_slot",
    values_to = "complication_type",
    values_drop_na = TRUE
  )

# Result:
# patient_id | complication_slot | complication_type
# 001        | complication_1    | bleeding
# 001        | complication_2    | infection
# 003        | complication_1    | thrombosis

The values_drop_na = TRUE argument is critical here — without it, you would keep all the NA rows, which represent the absence of a complication rather than a missing value. Patient 002 had no complications; we do not want three NA rows for them.

Going the other direction with pivot_wider() is useful when you need to create summary tables for reports:

# Create a cross-tabulation of complications by clinic
complication_summary <- complications_long %>%
  count(clinic_id, complication_type) %>%
  pivot_wider(
    names_from = complication_type,
    values_from = n,
    values_fill = 0
  )

Handling Missing Data

This is where clinical data work diverges most sharply from generic data science. In a Kaggle competition, you might impute missing values or drop incomplete rows without a second thought. In a clinical registry, every missing value has meaning, and how you handle it can affect the validity of your entire analysis.

The tidyr package gives you a few key tools:

# First: recode all the creative missing value indicators to proper NA
cleaned <- registry_data %>%
  mutate(across(everything(), ~case_when(
    . %in% c("-999", "-1", "N/A", "n.a.", "nicht angegeben", "") ~ NA_character_,
    TRUE ~ as.character(.)
  )))

# Replace NA with a specific value (e.g., for indicator variables)
cleaned <- cleaned %>%
  replace_na(list(
    complications_reported = "no",
    follow_up_completed = "unknown"
  ))

# Document missingness BEFORE dropping
missing_report <- cleaned %>%
  summarise(across(everything(), ~sum(is.na(.)))) %>%
  pivot_longer(everything(),
    names_to = "variable",
    values_to = "n_missing"
  ) %>%
  mutate(pct_missing = n_missing / nrow(cleaned) * 100) %>%
  filter(n_missing > 0) %>%
  arrange(desc(pct_missing))

# Only THEN drop rows for your specific analysis
analysis_set <- cleaned %>%
  drop_na(procedure_date, birth_date, sex)  # explicit about which vars

The rule I follow: never use drop_na() without first generating a missingness report. You need to know how many rows you are losing and from which variables. If 40% of your records are missing follow-up data, that is not a data cleaning problem — that is a finding about the registry's data collection process, and it belongs in your report.

I also avoid using drop_na() without arguments (which drops any row with any missing value in any column). In a dataset with 50 columns, this can silently eliminate the majority of your data. Always specify which columns are required for your particular analysis.

Interview Quick Review

What is dplyr/tidyr?

dplyr is an R package for data manipulation using a grammar of composable verbs (filter, mutate, group_by, summarise); tidyr is its companion for reshaping data between wide and long formats.

When would you use it?

Whenever you need to clean, transform, reshape, or summarize tabular data in R — especially clinical datasets with messy formats, missing values, and grouped structures that require aggregation.

Key function to remember:

df %>%
  group_by(group) %>%
  summarise(n = n(), mean = mean(x, na.rm = TRUE))

Common interview question:

"What is the difference between pivot_longer() and pivot_wider()?"

pivot_longer() converts columns into rows (wide to long format), while pivot_wider() converts rows into columns (long to wide format). Long format is usually needed for ggplot2 visualizations and statistical modeling, where each observation should be its own row. Wide format is more natural for human-readable summary tables and data entry forms.

One thing to remember:

Never silently drop missing clinical data. Always generate a missingness report before using drop_na(), and always specify which columns are required rather than dropping rows with any missing value.
