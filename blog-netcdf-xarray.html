Processing Large Scientific Datasets with Python and xarray

How xarray and NetCDF transformed my approach to handling multi-dimensional ocean model output during my postdoc.

During my postdoc at the University of Hamburg, I spent a significant portion of my time working with output from the ICON-O ocean model. A single simulation run could produce hundreds of gigabytes of data spread across thousands of files, each containing variables like temperature, salinity, and current velocity resolved across three spatial dimensions and time. Early on, I made the mistake of trying to wrangle this data with plain NumPy arrays and manual index bookkeeping. It was error-prone and painful. Switching to xarray changed everything.

The NetCDF Data Model

NetCDF (Network Common Data Form) is the lingua franca of atmospheric and ocean sciences. Unlike a CSV or a raw binary dump, a NetCDF file is self-describing: it carries its own metadata. Each file contains dimensions (e.g., time, depth, latitude, longitude), variables (the actual data arrays bound to those dimensions), and attributes (metadata like units, long names, and coordinate reference information).

A typical ocean model output file might have a structure like this:

# Inspecting a NetCDF file with ncdump (or Python)
import xarray as xr

ds = xr.open_dataset("ocean_output_2020.nc")
print(ds)

# Output:
# <xarray.Dataset>
# Dimensions:      (time: 365, depth: 40, lat: 180, lon: 360)
# Coordinates:
#   * time         (time) datetime64[ns] 2020-01-01 ... 2020-12-31
#   * depth        (depth) float64 5.0 15.0 25.0 ... 4500.0 5000.0
#   * lat          (lat) float64 -89.5 -88.5 ... 88.5 89.5
#   * lon          (lon) float64 0.5 1.5 2.5 ... 358.5 359.5
# Data variables:
#     temperature  (time, depth, lat, lon) float32 ...
#     salinity     (time, depth, lat, lon) float32 ...
#     u_velocity   (time, depth, lat, lon) float32 ...
#     v_velocity   (time, depth, lat, lon) float32 ...
# Attributes:
#     title:       ICON-O Global Ocean Simulation
#     institution: Max Planck Institute for Meteorology
#     history:     Created 2023-12-01

The beauty of this format is that the coordinates travel with the data. You never have to remember that index 12 along axis 1 corresponds to a depth of 200 meters -- the coordinate labels tell you directly. This is exactly the problem xarray was built to solve.

Working with xarray

xarray extends NumPy arrays with named dimensions and coordinate labels. Instead of writing data[45, 3, :, :] and hoping you got the axis order right, you write semantically meaningful selections. Here is a realistic workflow I used regularly when analyzing sea surface temperature patterns in the North Atlantic:

import xarray as xr
import matplotlib.pyplot as plt

# Open the dataset
ds = xr.open_dataset("ocean_output_2020.nc")

# Select sea surface temperature (top depth layer)
# .sel() uses label-based indexing -- no need to remember indices
sst = ds["temperature"].sel(depth=5.0)

# Extract a region: North Atlantic (40-65N, 60W-0E)
north_atlantic_sst = sst.sel(
    lat=slice(40, 65),
    lon=slice(300, 360)  # 300-360 = 60W to 0E in 0-360 convention
)

# Compute monthly mean SST across the region
monthly_mean = north_atlantic_sst.resample(time="1M").mean()

# Compute the spatial average for a time series
spatial_avg = monthly_mean.mean(dim=["lat", "lon"])

# Plot the result
spatial_avg.plot(figsize=(10, 4))
plt.title("Monthly Mean SST - North Atlantic (2020)")
plt.ylabel("Temperature (K)")
plt.tight_layout()
plt.savefig("north_atlantic_sst_2020.png", dpi=150)

Notice how every operation reads almost like plain English. The .sel() method selects by coordinate label, while .isel() selects by integer index if you need it. The .resample() method understands datetime coordinates natively, so computing monthly or seasonal averages is a one-liner. Compare this to the NumPy equivalent, where you would need to manually parse time stamps, compute bin edges, and loop over months -- all while carefully tracking axis positions.

Another pattern I used constantly was computing anomalies relative to a climatological mean:

# Compute the climatological monthly mean (average January, average February, etc.)
climatology = sst.groupby("time.month").mean(dim="time")

# Compute anomalies by subtracting the climatology
anomalies = sst.groupby("time.month") - climatology

# Find the month and location of the strongest positive anomaly
max_anomaly = anomalies.where(anomalies == anomalies.max(), drop=True)
print(f"Max anomaly: {float(anomalies.max()):.2f} K")
print(max_anomaly.coords)

The .groupby("time.month") operation automatically groups by calendar month and aligns the subtraction correctly. xarray handles the broadcasting behind the scenes. This kind of analysis, which is bread and butter in climate science, would require dozens of lines of careful index manipulation in raw NumPy.

Scaling with Dask

The single-file examples above work well when your data fits in memory. But ocean model output at high resolution can easily reach terabytes. During my work on the EERIE project, we dealt with ICON-O output at roughly 10 km horizontal resolution -- each time step was several gigabytes, and a multi-year simulation filled an entire tape archive.

The solution is lazy loading with Dask. By passing a chunks argument to xarray, the data is not loaded into memory until you explicitly request a computation. Instead, xarray builds a task graph that Dask executes in parallel:

import xarray as xr

# Open ALL NetCDF files matching a glob pattern
# chunks={"time": 100} means: load 100 time steps per chunk
ds = xr.open_mfdataset(
    "/work/bm1344/ocean_output/run_*/ocean_*.nc",
    chunks={"time": 100},
    parallel=True,
    combine="by_coords"
)

print(ds["temperature"])
# This prints immediately -- no data has been loaded yet
# <xarray.DataArray 'temperature' (time: 36500, depth: 40, lat: 1800, lon: 3600)>
# dask.array<shape=(36500, 40, 1800, 3600), dtype=float32, chunksize=(100, 40, 1800, 3600)>

# Compute the 100-year mean SST -- Dask processes this in chunks
mean_sst = ds["temperature"].sel(depth=5.0).mean(dim="time")

# .compute() triggers the actual parallel computation
result = mean_sst.compute()
result.to_netcdf("mean_sst_100yr.nc")

The key insight is that open_mfdataset with chunking turns your entire multi-file dataset into a single virtual array. You write the same xarray code as before, but under the hood Dask distributes the work across all available CPU cores (or across a SLURM cluster via dask-jobqueue). The .compute() call at the end is the only point where data actually flows through memory. This pattern let me process datasets that were orders of magnitude larger than my workstation's RAM without changing my analysis code at all.

Interview Quick Review

What is xarray?

xarray is a Python library that extends NumPy arrays with named dimensions and coordinate labels, making it natural to work with multi-dimensional scientific datasets stored in formats like NetCDF.

When would you use it?

Whenever you work with multi-dimensional labeled data -- climate model output, satellite imagery, time series of gridded observations -- especially when the data is stored in NetCDF or HDF5 format and may not fit in memory.

Key function to remember:

ds = xr.open_mfdataset("*.nc", chunks={"time": 100})

Common interview question:

"How do you handle datasets that don't fit in memory?"

Use Dask-backed xarray with chunked loading. By specifying a chunks parameter when opening datasets, xarray creates lazy arrays that are only computed on demand. Dask parallelizes the computation across chunks, so you can process terabyte-scale data on a machine with only a few gigabytes of RAM.

One thing to remember:

xarray's power comes from label-based indexing: .sel(depth=200, time="2020-06") is both more readable and less error-prone than data[150, 8, :, :]. Named dimensions eliminate an entire class of axis-ordering bugs.
